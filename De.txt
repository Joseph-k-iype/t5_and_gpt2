## 1. System Overview & Architecture

### 1.1 Purpose

This specification defines the full architecture for a production-grade system that converts Restriction and Condition nodes stored in FalkorDB into structured, machine-readable compliance rules. The system uses a swarm of domain-expert AI agents orchestrated by a supervisor, an MCP server for graph data retrieval, a React frontend for rule review and editing, and a test-validation loop before final promotion back to the graph.

### 1.2 End-to-End Flow

1. User opens the **Rules Dashboard (Page 1)**. Restrictions and Conditions are loaded grouped by jurisdiction/country with expandable tables showing linked context.
2. User clicks **"Convert to Rule"** on any Restriction or Condition. The UUID is submitted to the backend.
3. The **Supervisor Agent** launches the LangGraph swarm. A dedicated MCP tool queries FalkorDB for all context linked to that UUID — authorities, purposes, processes, data categories, subjects, sensitive categories, entities, jurisdictions — everything except DVCase nodes.
4. Expert agents process the context in parallel, each specialising in one dimension of the rule. Results are merged by the Synthesis Agent into a structured `MachineReadableRule` object.
5. The **React Flow Converter Agent** transforms the rule object into a React Flow node/edge graph specification.
6. **Page 2** renders the React Flow diagram. The user can edit rule properties, re-trigger synthesis, and iterate.
7. The **Test Generation Agent** produces business-readable test cases derived from the rule's trigger logic.
8. The user runs tests. A pass/fail report is shown. On failure the user can return to edit. On full pass, user clicks **"Approve & Push to Graph"**.
9. The approved rule is written back to FalkorDB and the source node is marked as converted.

> **Immutable Decision Rule:** Conditions map to `decision = PERMISSION`. Restrictions map to `decision = PROHIBITION`. This is enforced at the Supervisor Agent level before any sub-agent is invoked — it is never inferred by the model.

### 1.3 Technology Stack

| Layer | Technology | Purpose |
|---|---|---|
| LLM | `gpt-5-mini` (OpenAI) | All agent completions — no `temperature` or `max_tokens` params set |
| Agent Framework | LangGraph 0.2+ | Graph-based multi-agent orchestration with state + memory |
| Agent Memory | LangGraph MemorySaver / Checkpointer | Persist conversation and iteration state per rule UUID |
| Graph MCP | FastMCP (Python) | Expose FalkorDB openCypher queries as MCP tools |
| Backend API | FastAPI + Pydantic v2 | REST endpoints, SSE streaming, request/response validation |
| Frontend | React + Vite | SPA shell |
| State Mgmt | Zustand | Global app state (rules list, active rule, edit state, test results) |
| Animations | GSAP + Framer Motion | Page transitions, table reveals, diagram animations |
| Diagram | React Flow | Interactive rule logic graph on Page 2 |
| Styling | Tailwind CSS | Utility-first with custom CSS variables for theme |
| Graph DB | FalkorDB | Source of truth for all compliance graph data |

## 2. Machine-Readable Rule Schema

### 2.1 Overview

Every generated rule must conform to the `MachineReadableRule` Pydantic v2 schema. This schema is the single source of truth shared by backend, frontend, test generator, and the graph write-back module.

### 2.2 Top-Level Rule Object

| Field | Type | Required | Description |
|---|---|---|---|
| `ruleId` | UUID string | Yes | Auto-generated UUID. Stable across re-generations (derived from source node id). |
| `ruleTitle` | string | Yes | Title taken directly from the Restriction or Condition `name` field. |
| `description` | string | Yes | Agent-generated plain-English description of what this rule governs. |
| `decision` | enum: `PERMISSION` \| `PROHIBITION` | Yes | `PERMISSION` for Conditions. `PROHIBITION` for Restrictions. Set by Supervisor — never inferred. |
| `sourceNodeId` | UUID string | Yes | The Restriction or Condition UUID from the graph. |
| `sourceNodeType` | enum: `RESTRICTION` \| `CONDITION` | Yes | Derived from which relationship type connected the node to a DVCase. |
| `jurisdiction` | string | Yes | Jurisdiction label from the associated Country node. |
| `gbgf` | string | Yes | GBGF name this rule is scoped to. |
| `entityName` | string | Yes | Entity name this rule is scoped to. |
| `triggerLogic` | TriggerLogic object | Yes | Nested logical expression tree. See §2.3. |
| `actions` | list[Action] | Yes | One or more actions to be executed when the rule fires. |
| `output` | RuleOutput object | Yes | Describes the effect and downstream state after the rule fires. |
| `status` | enum: `DRAFT` \| `REVIEWED` \| `APPROVED` \| `PUSHED` | Yes | Lifecycle status. |
| `version` | integer | Yes | Incremented each time the rule is re-generated or edited. |
| `generatedAt` | ISO datetime | Yes | Timestamp of most recent generation. |
| `approvedAt` | ISO datetime | No | Set when user approves and pushes to graph. |

### 2.3 TriggerLogic — Nested Logical Expression

Trigger logic is expressed as a recursive tree of logical nodes. Agents are instructed to **prefer OR over AND** wherever semantically valid, reflecting that data transfer is generally allowed when any one of a set of conditions is met. The tree structure maps directly to the React Flow diagram.

| Field | Type | Description |
|---|---|---|
| `operator` | enum: `AND` \| `OR` \| `NOT` \| `LEAF` | The logical combinator for this node. `LEAF` nodes hold actual conditions. |
| `conditions` | list[TriggerLogic] | Child nodes. Empty for `LEAF` nodes. |
| `leafType` | enum: `PURPOSE` \| `PROCESS` \| `DATA_CATEGORY` \| `SUBJECT` \| `SENSITIVE_CATEGORY` \| `AUTHORITY` \| `JURISDICTION` \| `ENTITY` \| `CUSTOM` | Only set on `LEAF` nodes. |
| `leafId` | UUID string | Reference id of the linked graph node. Only set on `LEAF` nodes. |
| `leafLabel` | string | Human-readable label for the leaf. Only set on `LEAF` nodes. |
| `leafValue` | string | Specific value or qualifier for this condition. e.g. `IS_PRESENT`, `EQUALS`, `IN_LIST`. |
| `negated` | boolean | If true, wraps this node in NOT semantics. Default false. |

```json
// Example TriggerLogic structure:
{
  "operator": "OR",
  "conditions": [
    {
      "operator": "AND",
      "conditions": [
        { "operator": "LEAF", "leafType": "PURPOSE", "leafId": "uuid-1",
          "leafLabel": "Marketing", "leafValue": "IS_PRESENT" },
        { "operator": "LEAF", "leafType": "JURISDICTION", "leafId": "uuid-2",
          "leafLabel": "United Kingdom", "leafValue": "IS_RECEIVING" }
      ]
    },
    {
      "operator": "LEAF", "leafType": "SENSITIVE_CATEGORY", "leafId": "uuid-3",
      "leafLabel": "Health Data", "leafValue": "IS_ABSENT", "negated": true
    }
  ]
}
```

### 2.4 Action Object

| Field | Type | Description |
|---|---|---|
| `actionId` | UUID string | Auto-generated. |
| `actionType` | enum: `MASK_DATA` \| `ANONYMISE_DATA` \| `PSEUDONYMISE_DATA` \| `UPLOAD_EVIDENCE` \| `NOTIFY_DPO` \| `RESTRICT_TRANSFER` \| `REQUIRE_CONSENT` \| `APPLY_SAFEGUARD` \| `CUSTOM` | The operational action to execute. |
| `actionLabel` | string | Human-readable label for the action. |
| `actionDescription` | string | Agent-generated plain-English description of what this action does. |
| `duties` | list[Duty] | Duties that must be fulfilled in connection with this action. See §2.5. |

### 2.5 Duty Object

| Field | Type | Description |
|---|---|---|
| `dutyId` | UUID string | Auto-generated. |
| `dutyType` | enum: `CONSULT_TEAM` \| `OBTAIN_APPROVAL` \| `CONDUCT_REVIEW` \| `FILE_RECORD` \| `NOTIFY_AUTHORITY` \| `CUSTOM` | Type of duty. |
| `dutyLabel` | string | Human-readable label. |
| `dutyDescription` | string | Agent-generated description of what this duty requires. |
| `mandatory` | boolean | If true, this duty must be fulfilled for the action to be considered compliant. |

### 2.6 RuleOutput Object

| Field | Type | Description |
|---|---|---|
| `outputLabel` | string | Short label for the rule outcome. e.g. `Transfer Blocked`, `Transfer Permitted with Safeguards`. |
| `outputDescription` | string | Agent-generated plain-English description of the resulting state. |
| `downstreamEffect` | string | What happens downstream in the data transfer pipeline. |
| `graphAnnotation` | string | Label to write back to the Restriction/Condition node in FalkorDB on push. |

## 3. FalkorDB MCP Server

### 3.1 Purpose & Architecture

The MCP (Model Context Protocol) server is a lightweight FastMCP Python service that exposes FalkorDB as a set of typed tools callable by any LangGraph agent. It is the **exclusive data access layer** — no agent queries the graph directly. All queries are executed inside the MCP server via openCypher.

> **Note:** Queries are not specified in this document. Each MCP tool receives a node UUID and executes the appropriate openCypher traversal to return all connected data. The implementation team writes the queries during build; this spec defines only the tool names, inputs, outputs, and agent contracts.

### 3.2 MCP Tool Catalogue

| Tool Name | Input | Output Description | Used By |
|---|---|---|---|
| `get_restriction_full_context` | `restriction_id: str` | Complete context for a Restriction node: name, guidance, scoped GBGF, scoped Entity, jurisdiction Country, and all linked Purposes, Processes, DataCategories, DataSubjects, SensitiveCategories, Authorities, ReceivingJurisdictions. Excludes DVCase nodes. | Context Extraction Agent |
| `get_condition_full_context` | `condition_id: str` | Same structure as above but for Condition nodes. | Context Extraction Agent |
| `get_country_detail` | `country_id: str` | Full Country node properties including jurisdiction, region, countryCode. | Context Extraction Agent, Jurisdiction Agent |
| `get_purpose_detail` | `purpose_id: str` | Full Purpose node properties. | Context Extraction Agent, Purpose Agent |
| `get_process_detail` | `process_id: str` | Full Process node with hierarchy (level1/2/3). | Context Extraction Agent, Process Agent |
| `get_datacategory_detail` | `datacategory_id: str` | DataCategory with subcategories, sensitive categories, CDAHBIM mappings. | Context Extraction Agent, DataCategory Agent |
| `get_subject_detail` | `subject_id: str` | DataSubject node properties. | Context Extraction Agent, DataCategory Agent |
| `get_authority_detail` | `authority_id: str` | Authority node properties including countryCode. | Context Extraction Agent, Jurisdiction Agent |
| `get_gbgf_detail` | `gbgf_id: str` | Full GBGF node with level hierarchy. | Context Extraction Agent |
| `list_all_restrictions_by_jurisdiction` | `jurisdiction: str` | All Restriction nodes scoped to a given jurisdiction with name, id, converted status. | Frontend list endpoint |
| `list_all_conditions_by_jurisdiction` | `jurisdiction: str` | All Condition nodes scoped to a given jurisdiction with name, id, converted status. | Frontend list endpoint |
| `list_all_jurisdictions` | *(none)* | All distinct jurisdiction values present on Country nodes linked to Restrictions or Conditions. | Frontend list endpoint |
| `mark_rule_as_converted` | `node_id: str, rule_id: str, graph_annotation: str` | Sets the `converted_to_rule_id` and `rule_status` properties on the Restriction or Condition node. | Graph Write-Back Service |
| `write_rule_to_graph` | `rule: MachineReadableRule (JSON)` | Creates a MachineReadableRule node in the graph and links it to the source Restriction/Condition. | Graph Write-Back Service |

## 4. LangGraph Agent Architecture

### 4.1 Architecture Pattern: Swarm Hive with Supervisor

The system uses a **Swarm Hive** architecture in LangGraph where a Supervisor Agent acts as the central orchestrator. It dispatches work to specialist sub-agents, collects their outputs, resolves conflicts, and drives consensus. The swarm runs sub-agents in parallel where their inputs are independent, and sequentially where one agent's output feeds another.

### 4.2 Agent Registry

| Agent ID | Agent Name | Role | Runs |
|---|---|---|---|
| AGT-00 | Supervisor | Orchestrates the full swarm. Routes inputs. Resolves conflicts. Drives to `MachineReadableRule` output. Sets DECISION field — never delegates this. | Sequential controller |
| AGT-01 | Context Extraction Agent | Calls MCP to retrieve all graph context for the node UUID. Structures the raw response into a canonical `ContextBundle`. | First, always |
| AGT-02 | Restriction/Condition Analyst | Deep-reads the guidance text and all context. Produces semantic analysis: key obligations, prohibited actions, conditional triggers, and implicit constraints. | After AGT-01 |
| AGT-03 | Purpose & Process Domain Expert | Analyses purposes of processing and process hierarchy to determine which business activities are in or out of scope for this rule. | Parallel with AGT-04, AGT-05 |
| AGT-04 | Data Governance Expert | Analyses data categories, sensitive categories, subjects, and CDAHBIM classifications to determine data-layer constraints. | Parallel with AGT-03, AGT-05 |
| AGT-05 | Jurisdiction & Authority Expert | Analyses the jurisdiction, authority, country codes, and receiving entity landscape to determine geographic scope and regulator obligations. | Parallel with AGT-03, AGT-04 |
| AGT-06 | Rule Logic Synthesiser | Takes outputs of AGT-02 through 05. Builds the TriggerLogic tree using OR/AND/NOT operators. Prefers OR over AND. Maps all leaf nodes to graph UUIDs. | After AGT-03/04/05 |
| AGT-07 | Action & Duty Architect | Infers required actions and duties from the guidance text and context. | Parallel with AGT-06 |
| AGT-08 | Rule Validator | Checks the assembled rule for schema completeness, internal consistency, UUID coverage, logic contradictions, and DECISION alignment. | After AGT-06/07 |
| AGT-09 | React Flow Converter | Takes the validated `MachineReadableRule` and produces a React Flow node/edge specification. Assigns node types, positions, labels, and handle definitions. | After AGT-08 |
| AGT-10 | Test Case Generator | Reads the approved TriggerLogic and generates business-readable test cases: inputs that should and should not trigger the rule. | After user approval of rule |

### 4.3 LangGraph State Schema

All agents read from and write to a shared `TypedDict` state. The state is persisted by LangGraph's `MemorySaver` checkpointer, keyed on `thread_id = rule_uuid`.

```python
class RuleGenerationState(TypedDict):
    # Inputs
    source_node_id:        str          # UUID of Restriction or Condition node
    source_node_type:      str          # "RESTRICTION" or "CONDITION"

    # Context
    context_bundle:        dict         # Raw context returned by AGT-01 via MCP
    decision:              str          # "PERMISSION" or "PROHIBITION" — set by Supervisor, immutable

    # Agent outputs (each agent writes its own key)
    semantic_analysis:     dict         # AGT-02 output
    purpose_analysis:      dict         # AGT-03 output
    data_analysis:         dict         # AGT-04 output
    jurisdiction_analysis: dict         # AGT-05 output
    trigger_logic:         dict         # AGT-06 output (TriggerLogic tree)
    actions_and_duties:    dict         # AGT-07 output
    validation_result:     dict         # AGT-08 output: {passed: bool, issues: list}

    # Final outputs
    machine_readable_rule: dict         # Assembled MachineReadableRule (Pydantic-validated)
    react_flow_spec:       dict         # AGT-09 output: {nodes: [], edges: []}
    test_cases:            list         # AGT-10 output: [{id, description, inputs, expected}]
    test_results:          list         # Results after user runs tests

    # Lifecycle
    iteration_count:       int          # Number of regeneration iterations
    supervisor_notes:      list[str]    # Supervisor log messages
    errors:                list[str]    # Any errors during generation
```

### 4.4 LangGraph Graph Topology

```python
builder = StateGraph(RuleGenerationState)

builder.add_node("supervisor",             supervisor_agent)
builder.add_node("context_extraction",     context_extraction_agent)
builder.add_node("analyst",                restriction_analyst_agent)
builder.add_node("purpose_process",        purpose_process_agent)
builder.add_node("data_governance",        data_governance_agent)
builder.add_node("jurisdiction_authority", jurisdiction_authority_agent)
builder.add_node("logic_synthesiser",      logic_synthesiser_agent)
builder.add_node("action_duty_architect",  action_duty_architect_agent)
builder.add_node("rule_validator",         rule_validator_agent)
builder.add_node("react_flow_converter",   react_flow_converter_agent)
builder.add_node("test_case_generator",    test_case_generator_agent)

builder.set_entry_point("supervisor")
builder.add_edge("supervisor",         "context_extraction")
builder.add_edge("context_extraction", "analyst")

# Parallel fan-out
builder.add_edge("analyst", "purpose_process")
builder.add_edge("analyst", "data_governance")
builder.add_edge("analyst", "jurisdiction_authority")

# Fan-in to synthesiser and action architect
builder.add_edge("purpose_process",        "logic_synthesiser")
builder.add_edge("data_governance",        "logic_synthesiser")
builder.add_edge("jurisdiction_authority", "logic_synthesiser")
builder.add_edge("analyst",                "action_duty_architect")

# Validation with conditional retry
builder.add_edge("logic_synthesiser",    "rule_validator")
builder.add_edge("action_duty_architect","rule_validator")

builder.add_conditional_edges("rule_validator", route_after_validation, {
    "pass":  "react_flow_converter",
    "retry": "supervisor"            # Max 3 retry cycles
})

builder.add_edge("react_flow_converter", END)

# Test path triggered separately after user approval
builder.add_node("test_case_generator", test_case_generator_agent)
```

## 5. Agent Prompt Specifications

Each prompt is the agent's **system message**, written in structured markdown. All prompts use:
- **Chain of Thought (CoT):** explicit numbered reasoning steps the model must execute before producing output
- **Domain Expert Identity:** a specific, credible persona with named regulatory frameworks
- **Mixture of Experts (MoE):** the agent must evaluate the problem from multiple named perspectives before converging
- **Structured Output Contract:** a precise return format the orchestration layer can parse

> ⚠️ Curly-brace tokens (e.g. `{decision}`, `{guidance_text}`) are substituted at runtime by the orchestration layer before the message is sent to the model.

### 5.1 AGT-00 — Supervisor Agent

```markdown
# SUPERVISOR AGENT — Rule Generation Orchestrator

## Identity
You are the Master Orchestrator for a compliance rules conversion system at a major
global bank. Your role is to coordinate a swarm of specialist agents to convert a
single data transfer Restriction or Condition into a validated, machine-readable
compliance rule.

## Immutable Decision Assignment
The DECISION field is determined by node type and is NEVER inferred or changed by
any agent:
  - Source node type = RESTRICTION → decision = PROHIBITION
  - Source node type = CONDITION   → decision = PERMISSION

You MUST inject this value into state before dispatching any sub-agent.
  Node type: {source_node_type}
  Decision:  {decision}

## Orchestration Protocol
1. Confirm the DECISION assignment and log it to supervisor_notes.
2. Dispatch AGT-01 (Context Extraction). Await its ContextBundle.
3. Dispatch AGT-02 (Analyst) with the full ContextBundle.
4. Fan out AGT-03, AGT-04, AGT-05 in parallel once AGT-02 completes.
5. Await all three. Fan in to AGT-06 and AGT-07 simultaneously.
6. After AGT-06 and AGT-07: dispatch AGT-08 (Validator).
7. If validation fails: review the issue list. Provide targeted correction
   instructions to the relevant agent(s) and re-dispatch. Maximum 3 retry cycles.
8. If validation passes: dispatch AGT-09 (React Flow Converter). Return final state.

## Conflict Resolution
If parallel agents produce contradictory outputs, apply this priority:
  Jurisdiction Expert > Data Governance Expert > Purpose Expert
Regulatory constraints always override business-process framing.

## Output Protocol
Append a brief summary of each agent dispatch and its outcome to supervisor_notes[].
Log any retries with the reason and targeted correction instruction provided.
```

### 5.3 AGT-02 — Restriction/Condition Analyst

```markdown
# RESTRICTION & CONDITION ANALYST — Deep Semantic Analysis

## Identity
You are a Senior Privacy & Data Transfer Law Specialist with 20 years of experience
interpreting regulatory restrictions and conditions for cross-border data transfers at
tier-1 global financial institutions operating under GDPR, PDPO, CCPA, and LGPD.

## Chain of Thought Instruction
You MUST reason step by step through EVERY section below before producing output.
Do not skip reasoning steps. Label each step explicitly.

## Input
  Node name:      {source_node_name}
  Node type:      {source_node_type}
  Decision:       {decision}
  Guidance text:  {guidance_text}
  Full context:   {context_bundle}

## Reasoning Protocol — Execute ALL steps:

### Step 1: Surface Reading
Read the guidance text literally. What does it say in plain English?
What is the primary obligation or restriction stated?

### Step 2: Jurisdictional Lens
Consider the jurisdiction: {jurisdiction}.
What regulatory regime governs this jurisdiction?
Does the guidance align with GDPR Art. 46, PDPO, CCPA, or another framework?
How does the jurisdiction alter the interpretation?

### Step 3: Temporal & Conditional Analysis
Does the guidance contain conditions? (e.g. "only if", "provided that", "unless")
List all conditional clauses and what they gate.

### Step 4: Prohibition vs Permission Mapping
Decision: {decision}
If PROHIBITION: what specific data transfers, purposes, or processes are blocked?
If PERMISSION: what conditions must be met for the transfer to be allowed?

### Step 5: Implicit Constraints
What is implied but not stated? Consider:
  - Data minimisation requirements
  - Retention limits
  - Onward transfer restrictions
  - Controller/processor distinctions

### Step 6: Mixture of Experts — Three Perspectives
Interpret the guidance from three expert lenses and note where they diverge:
  A. Legal Compliance Officer perspective
  B. Data Architecture / Technical Controls perspective
  C. Business Operations / Risk Management perspective

## Output Format
Return a structured dict:
  primary_obligation:     str
  conditional_clauses:    list[str]
  blocked_elements:       list[{type, label, reason}]       // for PROHIBITION
  permission_conditions:  list[{type, label, condition}]    // for PERMISSION
  implicit_constraints:   list[str]
  regulatory_framework:   str
  expert_divergences:     list[str]
  confidence:             float  // 0.0 - 1.0
```

### 5.4 AGT-03 — Purpose & Process Domain Expert

```markdown
# PURPOSE & PROCESS DOMAIN EXPERT — Business Activity Scoping

## Identity
You are a Chief Data Officer with deep expertise in HSBC's Global Business and Global
Function taxonomy, business purpose classification, and process hierarchy governance.
You understand how GBGF codes, purpose taxonomies, and process level hierarchies
interconnect to define the scope of data processing activities.

## Chain of Thought Instruction
Work through each reasoning step explicitly and label it.

## Input
  Semantic analysis:  {semantic_analysis}
  Purposes:           {purposes}
  Processes:          {processes}  // with level 1/2/3 hierarchy
  GBGF:               {gbgf_name}  // Level: {gbgf_level}
  Decision:           {decision}

## Reasoning Protocol:

### Step 1: Purpose Scope Analysis
For each purpose:
  - Is it IN SCOPE (should trigger the rule)?
  - Is it OUT OF SCOPE (exclude or use as NOT condition)?
  - Confidence level for this classification?

### Step 2: Process Hierarchy Analysis
Examine each process at Level 1, Level 2, and Level 3:
  - Level 1: broad business activity categories
  - Level 2: functional sub-activities
  - Level 3: granular operational steps
  Which levels are directly implicated by the rule?

### Step 3: GBGF Alignment
Does the GBGF scope ({gbgf_name}) include or exclude any purposes or processes?
Are there processes at a different GBGF level that should be in scope?

### Step 4: Trigger Condition Formulation
For each in-scope purpose and process, formulate a trigger condition:
  - OR: any one of these purposes triggers the rule
  - AND: all must be present simultaneously
  Apply OR preference unless guidance explicitly requires co-occurrence.

### Step 5: MoE Cross-Check
  A. Maximally restrictive interpretation (minimum data transfer scope)
  B. Business enablement interpretation (maximise legitimate transfer scope)
  Document divergences. Use the more restrictive interpretation by default.

## Output Format
Return:
```

## 9. Test Runner Engine

### 9.1 Design Principle

The test runner evaluates each test case against the rule's `TriggerLogic` tree using a **pure Python recursive evaluator** — it does **not** call the LLM. The evaluator treats the TriggerLogic tree as a boolean expression tree and evaluates each `LEAF` condition against the test case's `active_leaf_ids` list (the set of UUIDs that are TRUE for that scenario).

### 9.2 Evaluation Engine

```python
# api/services/test_runner.py

def evaluate_trigger_logic(node: dict, inputs: dict) -> bool:
    """
    Recursively evaluates a TriggerLogic tree.
    inputs: { leafId: bool }  — True means the condition is active/present.
    """
    if node["operator"] == "LEAF":
        result = inputs.get(node["leafId"], False)
        return not result if node.get("negated") else result

    if node["operator"] == "AND":
        result = all(evaluate_trigger_logic(c, inputs) for c in node["conditions"])
    elif node["operator"] == "OR":
        result = any(evaluate_trigger_logic(c, inputs) for c in node["conditions"])
    elif node["operator"] == "NOT":
        result = not evaluate_trigger_logic(node["conditions"][0], inputs)
    else:
        raise ValueError(f"Unknown operator: {node['operator']}")

    return result


def run_test_case(test_case: TestCase, rule: MachineReadableRule) -> TestCaseResult:
    inputs = {leaf_id: True for leaf_id in test_case.active_leaf_ids}
    fired  = evaluate_trigger_logic(rule.triggerLogic.model_dump(), inputs)
    passed = (fired and test_case.expected_result == "FIRES") or \
             (not fired and test_case.expected_result == "DOES_NOT_FIRE")

    return TestCaseResult(
        test_id=test_case.id,
        passed=passed,
        actual_result="FIRES" if fired else "DOES_NOT_FIRE",
        expected_result=test_case.expected_result,
        error_explanation=None if passed else build_error_explanation(
            test_case, rule, inputs
        )
    )
```
